## 0. この文書の使い方

- 本書は、要件・設計ドキュメントを実装順に落とし込んだ **作業分解（WBS）**。
- 各 Phase は次を含む:
  - 目的
  - 完了条件（Definition of Done）
  - 詳細タスク（チェックリスト）
  - 主な成果物
  - 対応要件（FR/NFR）
- MVP を最短で成立させるため、**MVP必須** と **MVP後** を明示する。

## Phase 2: Supabase スキーマ・RLS・RPC 基盤

### 目的
- DB とセキュリティの土台を先に固定し、クライアント実装のブレを抑える。

### 完了条件（DoD）
- `posts / reactions / comments / profiles` の基本スキーマ作成済み。
- RLS が有効化され、本人操作制約が効いている。
- `get_local_feed`（必要なら `create_post`）RPC が動作し、5km + TTL 条件が担保される。

### タスク（DDL/RLS/RPC）
- [ ] `postgis` 拡張を有効化
- [ ] `profiles` テーブル定義（`auth.users` 連携方針を確定）
- [ ] `posts` テーブル作成（`location geography(Point, 4326)`, `expires_at` 含む）
- [ ] `reactions` テーブル作成（`UNIQUE(user_id, post_id)`）
- [ ] `comments` テーブル作成（`parent_comment_id` を許可）
- [ ] `idx_posts_location`（GiST）作成
- [ ] `expires_at` 初期設定方針決定（DB default か RPC 強制）
- [ ] RLS 有効化（全対象テーブル）
- [ ] 投稿更新・削除の本人制約ポリシー作成
- [ ] Storage バケット作成（投稿画像）
- [ ] Storage ポリシー（本人のみ更新/削除）作成
- [ ] `get_local_feed(lat, lng, limit, cursor, sort)` 実装
- [ ] `get_local_feed` の `ST_DWithin(..., 5000)` を実装
- [ ] `expires_at > now()` 条件をRPCに実装
- [ ] ページングカーソル仕様を確定
- [ ] （任意）`create_post` RPC を実装（`expires_at` 強制）

### タスク（検証）
- [ ] SQL 単体テスト（正常系/異常系）
- [ ] RLS で他人データ更新が拒否されることを検証
- [ ] 地理クエリの `EXPLAIN ANALYZE` を取得
- [ ] 想定データ件数でレスポンス確認

### 成果物
- マイグレーション一式
- RLS ポリシー定義
- RPC 定義と検証ログ

### 対応要件
- FR-POST-01/02/03
- FR-FEED-01/02/03
- FR-REACT-02
- NFR-SCALE-02
- NFR-SEC-01/02/03

## 付録A: MVP受け入れチェックリスト（抜粋）

- [ ] ログインできる（メール/OAuthのいずれか）
- [ ] 位置許可がない場合、ローカル機能が明確に制限される
- [ ] 投稿できる（テキスト必須、画像任意）
- [ ] 投稿位置はぼかし後のみ保存される
- [ ] 5km以内 + 期限内投稿のみフィード表示される
- [ ] フィードは新着順で表示される
- [ ] リアクション3種が付けられる（1ユーザー1投稿1つ）
- [ ] 主要画面の空/ロード/エラー状態がある
- [ ] RLSにより他人投稿の削除/更新ができない
- [ ] フィード応答と投稿反映が目標性能に概ね収まる

## 付録B: 推奨実装順（実装チーム向け）

1. Phase 1 -> 2 -> 3（基盤を先に固定）  
2. Phase 4 -> 5（認証と位置を先に安定化）  
3. Phase 6 -> 7 -> 8（MVP機能を縦に通す）  
4. Phase 9 -> 10（品質/非機能の仕上げ）  
5. リリース後に Phase 11/12 を段階投入
