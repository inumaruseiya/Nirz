## 1. はじめに

### 1.1 目的

本書は、ユーザーの現在地を基準に**半径5km以内**の投稿のみを閲覧・投稿できるローカルSNSについて、**Flutter クライアント**と **Supabase** を前提とした実装レベルの要件を定義する。  
「人」ではなく「場所・空気・今」を共有することを目的とし、**マッチング要素は持たない**。

### 1.2 背景・コンセプト

- ローカルの「今」を可視化する。
- ゆるい繋がり（匿名寄りの表示・**非マッチング**）。
- 短命な情報（リアルタイム性重視）。

### 1.3 対象読者

- プロダクトオーナー、エンジニア（Flutter / バックエンド）、QA、デザイナー。

### 1.4 参照・前提

- Supabase 公式ドキュメント（Auth, Database, Storage, RLS）。
- Flutter 公式ドキュメント、`geolocator` 等の位置情報プラグイン。
- PostGIS（`geography` 型、`ST_DWithin`、GiST インデックス）。

### 1.5 用語定義

| 用語 | 定義 |
|------|------|
| ローカルフィード | 閲覧者の現在地から半径5km以内かつ有効期限内の投稿一覧。 |
| ぼかし位置 | 端末で取得した正確座標にランダムオフセットを加え、サーバに保存する座標。正確座標は保存しない。 |
| TTL | 投稿の有効期限。本書では **24時間**（`expires_at`）。 |
| MVP | 最小実リリースに含める機能セット（後述）。 |

## 2. システム概要

### 2.1 システム構成（論理）

```
[Flutter アプリ]
    ├─ Supabase Auth（メール / OAuth）
    ├─ Supabase Database（Postgres + PostGIS, RLS）
    ├─ Supabase Storage（画像）
    └─ 位置: 端末取得 → クライアント側でオフセット → DB に geography として保存
```

### 2.2 スコープ

#### 2.2.1 含む

- 投稿（テキスト必須、画像任意）
- ローカルフィード（**半径5km固定**、有効期限内）
- リアクション（種類固定、1ユーザー1投稿1リアクション）
- コメント（スレッド、**ネスト1階層まで**）
- 位置ベース表示およびサーバ側のぼかし位置保存

#### 2.2.2 含まない（本書スコープ外）

- マッチング機能
- フォロー機能
- DM（初期は非対応）

## 4. 機能要件

### 4.1 認証（FR-AUTH）

| ID | 要件 |
|----|------|
| FR-AUTH-01 | メールまたは SNS（OAuth）によるログインを Supabase Auth で提供する。 |
| FR-AUTH-02 | 表示名は**ニックネーム**とし、匿名性を損なわない範囲でプロフィールを最小限（例: `name`, `avatar_url` は任意）とする。 |
| FR-AUTH-03 | 未ログイン時は投稿・リアクション・コメント等の書き込み操作を禁止する（RLS と整合）。 |

**Flutter 実装観点**

- `supabase_flutter` による `signInWithPassword` / OAuth フロー。
- セッション永続化とアプリ起動時のセッション復元。

---

### 4.2 位置情報（FR-LOC）

| ID | 要件 |
|----|------|
| FR-LOC-01 | 初回起動時（またはフィード利用前）に OS の位置情報許可をリクエストする。 |
| FR-LOC-02 | 端末から緯度・経度を取得し、**サーバには正確座標を送らない**。クライアントでぼかし処理した座標のみを `posts.location` 等に保存する。 |
| FR-LOC-03 | **位置ぼかし**: 取得点を基準に **±300m〜1km のランダムオフセット**（実装で距離分布を定義し、一様または設計した分布で生成）。 |
| FR-LOC-04 | **オフラインモード / 位置共有OFF**: 位置が取得できない、またはユーザーが共有を拒否した場合は、フィード取得を行わない、または明示メッセージを表示する（投稿作成は不可または別ポリシーで定義する場合は別紙で合意）。本書のデフォルトは「位置なしではローカルフィード・位置付き投稿は利用不可」。 |

**Flutter 実装観点**

- `geolocator`（または同等）で権限・精度取得。
- ぼかしは **WGS84 上でメートル換算のオフセット**（例: 方位角・距離をランダム化して目的地座標を算出）を推奨。サーバに生座標を送らないため、**投稿作成 RPC には既にぼかし後の lat/lng のみ**を渡す設計とする。

---

### 4.3 投稿（FR-POST）

#### 4.3.1 投稿内容

| 項目 | 必須 |
|------|------|
| テキスト `content` | 必須 |
| 画像 `image_url` | 任意（Storage アップロード後の公開URLまたはパス） |

#### 4.3.2 保存データ（論理）

| カラム | 型（概念） | 説明 |
|--------|------------|------|
| `id` | uuid | 主キー |
| `user_id` | uuid | 投稿者 |
| `content` | text | 本文 |
| `image_url` | text | 任意 |
| `location` | geography(Point) | **ぼかし後**の位置 |
| `created_at` | timestamptz | 作成時刻 |
| `expires_at` | timestamptz | `created_at + 24h` 等 |

#### 4.3.3 ビジネスルール

| ID | 要件 |
|----|------|
| FR-POST-01 | 投稿の有効期限は **24時間**（`expires_at`）。期限切れはフィードに出さない。 |
| FR-POST-02 | フィード上の「表示対象」は、**閲覧者の現在地**から半径 **5000m** 以内の投稿に限定する（閲覧者側座標はクエリに用いるが、サーバに正確座標を永続保存しない方針と整合させる）。 |
| FR-POST-03 | 画像は Supabase Storage にアップロードし、RLS / ポリシーで本人のみ削除・更新可能とする。 |

**Flutter 実装観点**

- 画像: `supabase_flutter` の Storage API または署名付きアップロード。
- 投稿 INSERT は `posts` 直接または **SECURITY DEFINER の RPC**（`expires_at` 自動設定をサーバ側で強制する場合）。

---

### 4.4 ローカルフィード（FR-FEED）

| ID | 要件 |
|----|------|
| FR-FEED-01 | 表示条件: 閲覧者のクエリ位置から **ST_DWithin(..., 5000)** かつ `expires_at > now()`。 |
| FR-FEED-02 | デフォルト並び: **新着順**（`created_at DESC`）。 |
| FR-FEED-03 | オプション並び: **人気順**（リアクション数集計。集計はビュー・マテビュー・サブクエリ等で実装）。 |

**参照クエリ（PostGIS）**

```sql
SELECT *
FROM posts
WHERE ST_DWithin(
  location,
  ST_MakePoint(:lng, :lat)::geography,
  5000
)
AND expires_at > NOW()
ORDER BY created_at DESC;
```

**Flutter 実装観点**

- フィード取得: Supabase の `rpc`（推奨: パラメータ検証・インデックス利用を固定）または PostgREST で地理条件付きクエリ（プロジェクトの PostgREST 制約に合わせて選択）。
- 一覧は `ListView` / `CustomScrollView`、ページング（`limit` / `cursor`）を非機能要件と合わせて設計。

---

### 4.5 リアクション（FR-REACT）

| ID | 要件 |
|----|------|
| FR-REACT-01 | 種類: **👍 / 👀 / 🔥** を `type` にマッピング（例: `like`, `look`, `fire` の enum 的制約）。 |
| FR-REACT-02 | **1ユーザー1投稿につき1リアクション**（DB で `UNIQUE (user_id, post_id)`）。変更は「更新」または「削除→追加」で統一。 |

---

### 4.6 コメント（FR-COMMENT）

| ID | 要件 |
|----|------|
| FR-COMMENT-01 | 投稿に紐づくスレッド形式。 |
| FR-COMMENT-02 | ネストは **1階層まで**（例: `parent_comment_id` NULL=トップレベル、子は `parent` のみ）。 |

---

### 4.7 ユーザーステータス（FR-STATUS・任意）

| ID | 要件 |
|----|------|
| FR-STATUS-01 | 「暇」「作業中」「外出中」等の軽量ステータスを保持してよい。 |
| FR-STATUS-02 | **ローカルフィードのカード一覧には表示しない**（プロフィール周り等の軽量UIのみ）。 |

---

### 4.8 モデレーション・通報・ブロック（FR-MOD）

| ID | 要件 |
|----|------|
| FR-MOD-01 | **NGワードフィルタ**（クライアント補助 + サーバ側推奨）。 |
| FR-MOD-02 | **通報** → 管理者確認フロー（テーブル `reports` 等は実装フェーズで定義）。 |
| FR-MOD-03 | **ユーザーブロック**（ブロック関係をテーブル化し、フィード・コメント取得から除外）。 |

> MVP では通報・ブロックを後回しにする場合、セクション11と整合させる。

## 5. 非機能要件

### 5.1 パフォーマンス（NFR-PERF）

| ID | 要件 |
|----|------|
| NFR-PERF-01 | フィード取得（代表エリア・想定件数）: **500ms 以内**を目標（サーバ処理 + ネットワーク。クライアント計測とサーバ APM で監視）。 |
| NFR-PERF-02 | 投稿反映（クライアント送信から一覧に自分の投稿が見えるまで）: **1秒以内**を目標。 |

### 5.2 スケーラビリティ（NFR-SCALE）

| ID | 要件 |
|----|------|
| NFR-SCALE-01 | Postgres + PostGIS を採用する。 |
| NFR-SCALE-02 | `posts.location` に **GiST インデックス**を必須とする。 |

```sql
CREATE INDEX idx_posts_location
ON posts
USING GIST (location);
```

### 5.3 セキュリティ（NFR-SEC）

| ID | 要件 |
|----|------|
| NFR-SEC-01 | 全該当テーブルで **RLS 有効化**。 |
| NFR-SEC-02 | 投稿の削除は **本人のみ**可能。 |
| NFR-SEC-03 | Storage オブジェクトにもポリシーを適用し、他ユーザーの上書き・削除を禁止。 |

### 5.4 プライバシー（NFR-PRIV）

| ID | 要件 |
|----|------|
| NFR-PRIV-01 | 正確な位置をサーバに保存しない（ぼかし仕様に従う）。 |
| NFR-PRIV-02 | 自宅等の特定を難しくするため、**毎回またはセッション単位でオフセット方針**を検討可能（実装詳細は設計書へ）。 |
| NFR-PRIV-03 | 位置共有OFF・オフライン時の挙動は FR-LOC-04 に従う。 |

### 5.5 可用性・運用

- Supabase のバックアップ・リージョン方針はインフラ設計に委ねる。
- 機能フラグ（コメント・通報の段階リリース）を推奨。

## 6. データベース設計（Supabase）

### 6.1 テーブル定義（論理）

#### `users`（public プロフィール拡張の例）

アプリが `auth.users` と同期する `public.profiles` を置く構成が一般的。本書の「最小限」に合わせ、概念として以下を保持する。

| カラム | 型 | 説明 |
|--------|-----|------|
| `id` | uuid | `auth.users.id` と同一 |
| `name` | text | ニックネーム |
| `created_at` | timestamptz | 作成 |

（実装では `profiles` 名・追加列はマイグレーションで定義。）

#### `posts`

| カラム | 型 | 説明 |
|--------|-----|------|
| `id` | uuid | PK |
| `user_id` | uuid | FK |
| `content` | text | 必須 |
| `image_url` | text | 任意 |
| `location` | geography(Point, 4326) | ぼかし後 |
| `created_at` | timestamptz | |
| `expires_at` | timestamptz | TTL |

#### `reactions`

| カラム | 型 | 説明 |
|--------|-----|------|
| `id` | uuid | PK |
| `user_id` | uuid | |
| `post_id` | uuid | |
| `type` | text | `like` / `look` / `fire` 等 |
| `created_at` | timestamptz | |

制約: `UNIQUE (user_id, post_id)`。

#### `comments`

| カラム | 型 | 説明 |
|--------|-----|------|
| `id` | uuid | PK |
| `user_id` | uuid | |
| `post_id` | uuid | |
| `parent_comment_id` | uuid | NULL 許容（1階層ネスト用） |
| `content` | text | |
| `created_at` | timestamptz | |

### 6.2 RLS 方針（概要）

- `SELECT`: 認証ユーザーに対し、**地理条件・有効期限はクエリ側**で満たす。テーブルポリシーは「公開読取り禁止・RPC のみ」または「認証済み read 可 + アプリで絞り込み」から選択。**推奨は `get_local_feed(lat, lng)` RPC + SECURITY DEFINER で一箇所に集約**。
- `INSERT` / `UPDATE`: 認証ユーザーは自分の `user_id` のみ。
- `DELETE`: 投稿は本人のみ。

（詳細ポリシーは別紙「セキュリティ設計」に記載推奨。）

## 7. API・インターフェース設計

### 7.1 REST 的エンドポイント（概念）

PostgREST / Supabase Client ではテーブル名・RPC 名にマッピングされる。

| 操作 | 方法 | パス（概念） | 備考 |
|------|------|----------------|------|
| 投稿作成 | POST | `posts` または `rpc/create_post` | `expires_at` サーバ設定推奨 |
| フィード取得 | GET / RPC | `posts?...` または `rpc/get_local_feed` | lat/lng はクエリパラメータ |
| リアクション | POST / UPSERT | `reactions` | UNIQUE 制約と整合 |
| コメント | POST | `comments` | |

### 7.2 Flutter（クライアント）からの呼び出し

- `Supabase.instance.client.from('posts').insert(...)` 等。
- 地理検索は **RPC 呼び出し** `client.rpc('get_local_feed', params: {...})` を推奨。
- リアルタイム更新が必要な場合は `supabase_flutter` の Realtime 購読をオプションで検討（MVP はポーリングでも可）。

## 9. MVP 定義

### 9.1 MVP に含める（必須）

- 認証（メール or SNS）
- 投稿（テキスト＋任意画像、ぼかし位置、24h TTL）
- **5km フィード**（新着順）
- リアクション（3種、1ユーザー1投稿1リアクション）

### 9.2 MVP 後（後回し可）

- コメント
- 通報・ブロック・管理者フロー
- ユーザーステータス
- フィードの人気順オプション（実装コストに応じて MVP に前倒し可）

## 10. リスクと対策

| リスク | 内容 | 対策 |
|--------|------|------|
| 過疎 | 表示件数が少ない | 初期は**エリア限定リリース**、シード運用、通知は慎重に。 |
| 治安 | 悪用・トラブル | **通報・ブロック**を早期に実装（MVP 後でもロードマップ固定）。 |
| 位置プライバシー | 特定追跡 | ぼかし必須、正確座標非保存、プライバシーポリシー整備。 |
| パフォーマンス | 地理クエリの負荷 | GiST、RPC 集約、ページング、`EXPLAIN ANALYZE` による検証。 |
